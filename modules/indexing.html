Indexing — from O(n) scans to O(log n) lookups

An index is a small, sorted side-structure of (key → pointer) entries that lets the database jump to a row without scanning the whole table. We'll build up from a brute-force scan to single-level ordered indexes (dense & sparse), then multilevel indexes, and finally the B+-tree — the balanced structure real databases use.

5Pitfalls & exam tips

⚠️ Common mistakes

  • Over-indexing: every index speeds reads but slows INSERT/UPDATE/DELETE (the index must be maintained) and costs storage.
  • A sparse index requires the data file to be physically ordered on the key; a dense index doesn't.
  • Indexing a low-selectivity column (e.g. a boolean) rarely helps — the optimizer may still scan.
  • Functions on an indexed column in WHERE (e.g. WHERE UPPER(name)=…) can disable the index.

💡 Exam tips

  • Primary index = on the ordering key (often sparse); secondary index = on a non-ordering field (must be dense).
  • Multilevel: each level reduces the search by the fan-out f, so cost ≈ ⌈logf(blocks)⌉.
  • B+-trees keep all data in the leaves and link them, so range queries are very efficient.
  • B+-trees stay balanced — every search visits the same number of levels (root → leaf).
