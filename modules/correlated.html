Correlated Subqueries — EXISTS / NOT EXISTS

A correlated subquery references a column from the outer query, so it can't be evaluated once — it re-runs for every outer row. EXISTS simply asks "did the inner query return any row?" and returns true/false. It's the natural tool for "customers who have ordered" and, with NOT EXISTS, "customers who never ordered."

5Pitfalls & exam tips

⚠️ Common mistakes

  • Forgetting the correlation condition (WHERE o.cust_id = c.id) — without it EXISTS is true for every outer row.
  • Selecting real columns inside EXISTS — it only cares whether a row exists, so SELECT 1 is idiomatic.
  • Reaching for NOT IN when NULLs are possible — use NOT EXISTS, which is NULL-safe.

💡 Exam tips

  • Read EXISTS as: "for this outer row, does at least one matching inner row exist?"
  • NOT EXISTS is the standard way to express "for all / none" (relational division).
  • EXISTS can stop at the first matching row, so it's often efficient despite re-running.
  • EXISTS handles NULLs correctly where NOT IN silently fails.
