Triggers — code that fires on data changes

A trigger is procedural code the database runs automatically in response to an INSERT, UPDATE, or DELETE on a table. It follows the Event → Condition → Action model: an event fires it, an optional condition gates it, and an action runs. Triggers enforce rules, keep audit logs, and maintain derived data — without the application having to remember.

5Pitfalls & exam tips

⚠️ Common mistakes

  • Referencing NEW in a DELETE trigger or OLD in an INSERT trigger — those pseudo-rows don't exist there.
  • Trying to modify NEW in an AFTER trigger — too late; the row is already written. Modify it in BEFORE.
  • Triggers that update the same table can cascade and even recurse — easy to cause loops.
  • Hidden business logic in triggers can be hard to debug; document them well.

💡 Exam tips

  • Remember E-C-A: Event fires it, Condition (WHEN) gates it, Action runs.
  • INSERT → only NEW; DELETE → only OLD; UPDATE → both.
  • BEFORE = validate/modify the incoming row; AFTER = log/cascade once it's stored.
  • FOR EACH ROW fires N times for an N-row statement; statement-level fires once.
