Views — virtual tables built from a query

A view is a named, stored SELECT statement. You query it exactly like a table, but it holds no data of its own — every time you read it, the database runs the underlying query against the live base tables. Views simplify complex queries, hide sensitive columns, and give each user a tailored window onto the data.

5Pitfalls & exam tips

⚠️ Common mistakes

  • Thinking a view stores rows — it doesn't; it re-executes its query on every access.
  • Trying to INSERT/UPDATE a view built on joins, GROUP BY, DISTINCT, or aggregates — generally not updatable.
  • Expecting a materialized view to be current — it's a snapshot until you REFRESH it.
  • Heavy views on views on views — each layer re-runs, which can get slow.

💡 Exam tips

  • Define a view: CREATE VIEW v AS SELECT …; — query it like any table.
  • Views provide logical data independence and security (expose only chosen columns/rows).
  • A simple view (one table, no aggregation) is usually updatable; changes pass to the base table.
  • WITH CHECK OPTION stops updates that would make a row vanish from the view.
