Finding Keys from Functional Dependencies

A superkey is any attribute set whose closure is the whole relation. A candidate key is a minimal superkey — drop any attribute and it stops determining everything. We'll classify attributes by where they appear in the FDs to shrink the search, then compute closures to pin down every candidate key, and finally label attributes prime or non-prime — the vocabulary normalization needs.

5Pitfalls & exam tips

⚠️ Common mistakes

  • Calling any superkey a candidate key — candidate keys must be minimal.
  • Forgetting a relation can have several candidate keys (e.g. both C and AB here).
  • Missing that attributes appearing on no right-hand side must be in every candidate key.
  • Assuming one attribute is enough — always verify with the closure that it determines all of R.

💡 Exam tips

  • Attributes that never appear on a right side are essential — in every key. Right-only attributes are in no key.
  • Start the search from the essential set; add "both-side" attributes until the closure covers R.
  • Check minimality: remove each attribute and confirm the closure is no longer everything.
  • Prime/non-prime is decided after you know all candidate keys — it drives 2NF and 3NF.
